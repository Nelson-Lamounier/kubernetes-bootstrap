# EC2 Image Builder Component YAML Exceeds 16,000-Byte AWS Limit

## Symptom

Adding a new step to the Golden AMI EC2 Image Builder component has no effect — the AMI does not re-bake, the CDK deployment completes without error, and CloudFormation reports the stack as `UPDATE_COMPLETE`. The old AMI ID remains in SSM and new EC2 instances continue launching from the previous AMI.

There is no CloudFormation error. No Image Builder pipeline failure is logged. The change is silently dropped.

## Root cause

AWS imposes a hard limit of **16,000 characters** on the `Data` field of an `AWS::ImageBuilder::Component` resource. When the generated YAML exceeds this limit, CloudFormation silently rejects the update to the component — the resource is not replaced, the Image Builder recipe version does not change, and no new AMI bake is triggered.

The component YAML is generated by `buildGoldenAmiComponent()` in `infra/lib/constructs/compute/build-golden-ami-component.ts`. The file header records this constraint:

```typescript
// Comments are intentionally stripped from the generated YAML to stay under
// the 16 000-character EC2 Image Builder component limit. Explanations live
// here in the TypeScript source.
```

The hash prefix added to the YAML (for content-addressed invalidation) is compacted to `#<hash>\n` (6 bytes + hash length) rather than the previous `# scripts-hash: <hash>\n` (30+ bytes) to preserve headroom.

Source: commits `eb2e35b` and `3e11116`.

## How to diagnose

```bash
# 1. Synthesise the stack and measure the component Data field length
cd infra
npx cdk synth --quiet 2>/dev/null \
  | python3 -c "
import sys, json, yaml
template = yaml.safe_load(sys.stdin.read())
for k, v in template.get('Resources', {}).items():
    if v.get('Type') == 'AWS::ImageBuilder::Component':
        data = v.get('Properties', {}).get('Data', '')
        print(f'{k}: {len(data)} chars (limit=16000, headroom={16000-len(data)})')
"
# Expected: headroom > 0
# If headroom < 0: the update will be silently dropped
```

## How to fix

### Reduce YAML size to get below the limit

The `buildGoldenAmiComponent()` function generates the YAML. Candidates for trimming without behaviour change:

- Shorten step `description:` fields — these appear in the AWS console only
- Remove `echo` diagnostic lines that duplicate information already in logs
- Combine related shell commands into a single `commands:` entry to reduce YAML structure overhead
- Remove inline validation commands that duplicate the `validate` phase

After trimming, re-run the diagnostic above to confirm headroom.

### Split into multiple components (structural fix)

If the component is approaching the limit regularly, split it into two `AWS::ImageBuilder::Component` resources and chain them in the Image Builder recipe. The `GoldenAmiImageConstruct` in `infra/lib/constructs/compute/golden-ami-image.ts` accepts an array of component ARNs.

This is a larger CDK change but eliminates the size constraint permanently.

## How to prevent

Add a size check to the CDK synthesis:

```typescript
// In buildGoldenAmiComponent() or its caller:
const componentYaml = buildGoldenAmiComponent(input);
if (componentYaml.length > 15_800) {  // 200-char safety margin
  throw new Error(
    `Image Builder component YAML is ${componentYaml.length} chars ` +
    `(limit=16000, headroom=${16000 - componentYaml.length}). ` +
    `Trim or split the component before deploying.`
  );
}
```

The `200-char` margin accounts for the hash prefix that `buildGoldenAmiComponent` prepends. Without this guard, engineers adding new AMI steps will not discover the limit until observing that their AMI change silently had no effect.
