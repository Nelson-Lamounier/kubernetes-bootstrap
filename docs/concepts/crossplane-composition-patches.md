# Crossplane Composition Patches

How `FromCompositeFieldPath` patches map claim parameters to managed resource fields — the `Format` string transform used for external naming, the `bucketSelector.matchLabels` cross-resource reference pattern, array index notation in `toFieldPath`, and what happens when a patch source field is absent.

## What patches do

A Composition patch is a single declarative rule of the form "copy field A from the Composite Resource into field B of this managed resource, optionally transforming the value." Patches are the mechanism that connects the developer-facing claim parameters to the AWS-provider managed resource specification.

Without patches, every managed resource would be static — every bucket would have the same name, the same retention, and the same tags regardless of what the developer specified. Patches are what make Compositions reusable.

## FromCompositeFieldPath — the primary patch type

All patches in this cluster's Compositions use `type: FromCompositeFieldPath`. It reads a value from the Composite Resource (which the Crossplane engine populates from the claim's `spec.parameters`) and writes it into the managed resource's `spec.forProvider`.

Basic form:

```yaml
patches:
  - type: FromCompositeFieldPath
    fromFieldPath: spec.parameters.retentionDays
    toFieldPath: spec.forProvider.rule[0].expiration[0].days
```

`fromFieldPath` is a dot-notation path into the Composite Resource object. `toFieldPath` is a dot-notation path into the managed resource. The Crossplane engine evaluates this on every reconcile loop — if the composite field changes, the managed resource field is updated.

## Format transform — adding the naming prefix

The `crossplane-shared-` prefix on all AWS resource names is enforced via a `string.Format` transform on the `crossplane.io/external-name` annotation patch:

```yaml
patches:
  - type: FromCompositeFieldPath
    fromFieldPath: spec.parameters.bucketName
    toFieldPath: metadata.annotations[crossplane.io/external-name]
    transforms:
      - type: string
        string:
          type: Format
          fmt: "crossplane-shared-%s"
```

The `%s` receives the value from `fromFieldPath` after all transforms are applied left-to-right. A claim with `bucketName: my-assets` produces an annotation value of `crossplane-shared-my-assets`, which the S3 provider uses as the actual AWS bucket name.

The DLQ in `XMonitoredQueue` uses the same pattern with a `-dlq` suffix:

```yaml
transforms:
  - type: string
    string:
      type: Format
      fmt: "crossplane-shared-%s-dlq"
```

`crossplane.io/external-name` is a standard Crossplane annotation. When set on a managed resource, it overrides the default external name (which would otherwise be the Kubernetes object name). This is required here because the Kubernetes object name is auto-generated by the Composition engine — using the external-name annotation is the only way to control what name appears in AWS.

## Array index notation in toFieldPath

`toFieldPath` supports array index notation to target a specific element in a list:

```yaml
toFieldPath: spec.forProvider.rule[0].expiration[0].days
```

This navigates into `spec.forProvider.rule`, takes the first element (`[0]`), navigates into its `expiration` field, takes the first element, and sets `days`. The base template for the lifecycle resource already defines the array structure with a single element (id `auto-expire`); the patch overwrites only the `days` value.

If the array does not exist at the path, Crossplane creates it. If the array exists and the index is out of bounds, the behavior depends on Crossplane version — for existing compositions, ensure the base template pre-declares the array.

## bucketSelector — cross-resource reference via label matching

The four sub-resources in `XEncryptedBucket` (`BucketVersioning`, `BucketServerSideEncryptionConfiguration`, `BucketPublicAccessBlock`, `BucketLifecycleConfiguration`) each need to reference the parent `Bucket` resource to associate themselves with it at the AWS API level. They use a `bucketSelector` field rather than a direct name reference:

```yaml
patches:
  - type: FromCompositeFieldPath
    fromFieldPath: spec.parameters.bucketName
    toFieldPath: spec.forProvider.bucketSelector.matchLabels[crossplane.io/claim-name]
```

This patches the claim name into the sub-resource's selector. The `provider-aws-s3` controller resolves `bucketSelector.matchLabels` to find the `Bucket` managed resource with a matching `crossplane.io/claim-name` label, then uses that bucket's external name (the AWS bucket name) to make the sub-resource API call.

The pattern is: **sub-resources reference their parent by claim label, not by hardcoded name.** This makes the Composition reusable — the same Composition template works for any claim name because the selector is patched dynamically per claim.

## Tags patch — scalar value copy

Copying the `environment` parameter into a tags map is a plain scalar copy (no transforms):

```yaml
patches:
  - type: FromCompositeFieldPath
    fromFieldPath: spec.parameters.environment
    toFieldPath: spec.forProvider.tags.environment
```

This adds (or overwrites) the `environment` key in the managed resource's tags map. The base template pre-declares other tags (`managed-by: crossplane`, `project: cdk-monitoring`) — the patch only updates `environment` without affecting the other tags.

## What happens when a patch source field is absent

When `fromFieldPath` points to a field that does not exist in the Composite Resource (e.g. an optional parameter the developer did not set), the patch is silently skipped — the managed resource field retains its base template value. This is the default policy: `policy.fromFieldPath: Optional`.

To make a patch required (fail reconciliation if the source field is absent):

```yaml
policy:
  fromFieldPath: Required
```

Neither of the current Compositions uses `Required` policy — all patched parameters have defaults in the XRD schema (`retentionDays: 90`, `visibilityTimeoutSeconds: 30`, `environment: development`), so the source fields are always present after claim admission.

## What the Compositions do not patch (platform defaults)

Platform-enforced values that developers cannot override via claim parameters:

| Value | Resource | Why it is hardcoded |
|-------|----------|-------------------|
| `sseAlgorithm: AES256` | BucketServerSideEncryptionConfiguration | Encryption algorithm is a platform security requirement |
| `blockPublicAcls: true` (and 3 others) | BucketPublicAccessBlock | Public access is never permitted |
| `status: Enabled` | BucketVersioning | Versioning cannot be disabled |
| `sqsManagedSseEnabled: true` | Both SQS queues | Encryption at rest is always on |
| `messageRetentionSeconds: 1209600` | DLQ | 14-day DLQ retention is a platform default; developers cannot reduce it |
| `region: eu-west-1` | All resources | Single-region deployment; no multi-region support |

These defaults are embedded in the `base:` section of each Composition resource — they are not exposed as XRD parameters and therefore not patchable from claims.
